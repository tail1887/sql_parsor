# 06. 4분 발표 대본 (A/B 비교 버전)

## 1. 서론

안녕하세요. 이번에 2조의 발표를 맡게 된 박태정입니다.

이번 프로젝트는 2인 1조 페어 프로그래밍으로 진행되었는데요. 설명의 편의를 위해 각 팀을 A조와 B조로 지칭하여 말씀드리겠습니다.

우선 A조는 Lexer, Parser 등 시스템 구성 요소를 코드 단위로 순서대로 개발하며, 전체적인 실행 흐름과 시스템 구조를 파악하는 'Bottom-up' 방식에 집중했고.

반면 B조는 실제 다룰 데이터 테이블을 먼저 정의한 뒤, 이를 처리하기 위한 기능을 우선 구현하는 'Use-case' 중심으로 접근했습니다.

## 2. 전체 아키텍처

본격적인 설명에 앞서 전체 아키텍처를 살펴보겠습니다.

흐름은 이렇습니다. CLI가 SQL 파일을 읽어 문장 단위로 분리하면, Lexer가 토큰으로 쪼개고, Parser가 구문 분석을 통해 AST(추상 구문 트리)를 생성합니다. 마지막으로 Executor가 이 트리 정보를 바탕으로 실제 동작을 수행하며, 그 결과는 터미널 출력과 데이터 파일 저장으로 나타납니다.

## 3. 계층별 상세 설명: (1) CLI 계층

이제 계층별 특징을 조별로 비교하며 설명하겠습니다. 

첫 번째는 시작점인 CLI 계층입니다.  
main 함수에서 파일 경로를 받으면 sql_processor가 문장 단위로 렉서에게 넘겨주는데, 저희는 여기서 에러 처리 방식에 차이가 있었습니다.

A조는 에러를 '파싱 오류(2)'와 '실행/IO 오류(3)'로 세분화하여 종료 코드를 관리했습니다. 덕분에 에러 원인을 빠르게 파악하기 유리한 장점이 있었는데요.

반면에 B조는 사용자에게 메시지는 상세히 보여주되, 종료 코드는 '실패(1)'로 단일화하여 처리 흐름을 단순화했습니다.

## 4. 계층별 상세 설명: (2) Lexer (Tokenizer)

다음은 문자열을 토큰 단위로 분해하는 Lexer입니다.

A조는 상태를 유지하며 요청이 있을 때마다 토큰을 하나씩 반환하는 lexer_next() 방식을 채택했습니다. 이는 파서가 필요한 만큼만 렉서와 단계적으로 연결되어서, 결합이 느슨하고 확장·제어하기 쉽다는 장점이 있었고요.

그리고 B조는 문장 전체의 토큰을 TokenList로 모두 만든 뒤 파서에 넘깁니다. 이 방식은 디버깅 시 전체 토큰 내역을 한눈에 확인하기 좋고, 파서 구현이 단순해진다는 장점이 있었습니다.

## 5. 계층별 상세 설명: (3) Parser

다음으로 토큰이 문법에 맞는지 검증하고 AST를 생성하는 Parser입니다.

A조는 일반적인 SELECT, INSERT 문법을 폭넓게 처리할 수 있도록 설계하여 향후 확장성이 뛰어났고요.

B조는 특정 테이블(STUDENT, ENTRY_LOG)을 처리하기 위한 5가지 패턴에 집중한 전용 파서를 구현했습니다. 덕분에 고정된 요구사항을 매우 안정적으로 처리한다는 장점이 있었습니다.

## 6. 계층별 상세 설명: (4) Executor & Storage

다음으로 실제 동작을 수행하는 Executor와 파일 입출력을 담당하는 Storage입니다.
Parser가 "의미"를 만든다면, Executor는 그 의미를 "동작"으로 바꿉니다.

A조는 Executor가 Parser의 결과를 Storage에 바로 연결하는 심플한 구조를 가졌습니다. 데이터 저장 역시 CSV 하나에 집중했고요.

B조는 Executor 내부에서 권한 체크, 데이터 타입 변환(Datetime), 존재 여부 검사 등 복잡한 비즈니스 로직을 함께 처리합니다. 또한 CSV뿐 아니라 Binary 로그 파일 등 매체도 다양화했습니다.

한 줄로 요약하자면 A조는 간결한 파이프라인을, B조는 엄격한 도메인 규칙을 중시했습니다.

## 7. 실행 결과 및 결론

마지막으로 실행 결과입니다. 이 부분은 실시간 터미널 출력과 data 폴더 내의 파일 저장으로 확인 가능한데요.

A조는 결과를 TSV 형태로 깔끔하게 시각화하여 처리 과정을 직관적으로 확인하는 데 주력했고,

B조는 단순 출력을 넘어 Binary 로그 조회 및 데이터 정합성 검증까지 포함하여, 엄격한 데이터 관리에 집중했습니다.

---

데모 시연은 시간 관계상, 먼저 B조의 데이터 설계 포인트를 짧게 보고, 이어서 A조 아키텍처로 전체 실행 흐름을 검증하겠습니다.

B조는 SQL 처리기를 활용해 302호 출입 관리 시나리오를 구현했고, 데이터 성격에 따라 저장 방식을 분리했습니다.  
학생 기본 정보와 권한처럼 사람이 자주 확인하는 기준 데이터는 CSV로 저장했고,  
출입 기록처럼 계속 누적되는 이벤트 데이터는 고정 길이 Binary append-only 구조로 저장했습니다. 다만 SELECT 시에는 Binary 데이터를 사람이 읽기 쉬운 문자열로 변환해서 출력합니다.

**A조는 파이프라인이 단순하고, 데모 페이지에 CLI를 연동해 실행 과정을 단계별로 확인하기 좋은데요.**

**데모 페이지에서 SQL을 실행하면, Lexer 토큰, Parser AST, Executor 호출 흐름, 그리고 stdout/exit code/CSV diff를 순서대로 확인할 수 있습니다.**

마지막으로 `users.csv` 파일을 확인하면,  
화면에서 본 결과와 실제 파일 반영이 일치하는 것을 확인할 수 있습니다.

정리하면,  
A조는 전체 파이프라인의 가시성과 검증 편의성을,

그리고 B조는 규칙 중심 설계의 강점을 보여줬습니다.
