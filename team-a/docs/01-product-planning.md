# 01. Product Planning

이 문서는 SQL 처리기 프로젝트의 범위와 성공 기준을 정의합니다.

## 1) 프로젝트 한 줄 소개

- 프로젝트명: **sql_parsor** (SQL Processor)
- 한 줄 설명: 텍스트 파일에 적은 **INSERT / SELECT** 문을 CLI로 읽어 **파싱·실행**하고, **테이블당 파일(CSV)** 에 저장·조회하는 **C 기반 미니 DB 엔진**
- 개발 기간: 수요 코딩회(1일 몰입) + 마감·발표(목요일 오전)
- 팀 구성: (팀원 역할 기입)

## 2) 문제 정의

- 문제 1: SQL이 **텍스트**로 주어질 때, 이를 **구조화**하고 **파일 기반 저장소**에 반영하는 **처리기 파이프라인**을 직접 구현해 볼 필요가 있다.
- 문제 2: DB 내부의 **파싱 → 실행 → 저장** 흐름을 코드로 경험해야 **이력서·포트폴리오** 수준의 설명이 가능하다.
- 기존 방식의 한계: 상용 DB나 고수준 라이브러리에만 의존하면 **lexer/parser/실행기**의 경계가 흐려진다.

## 3) 타겟 사용자

- 핵심 사용자: **개발자(본인·팀)** — 로컬에서 `.sql` 스크립트로 데이터를 적재·조회한다.
- 사용 상황: 과제 제출, 데모 발표(README 기준 4분 + Q&A 1분), 포트폴리오 저장소.
- 사용 빈도/맥락: CLI로 **한 번에 한 파일** 실행(배치 스크립트).

## 4) 목표

- 제품 목표: **CLI + SQL 파일 입력 + INSERT/SELECT + 파일 DB** 가 동작하는 **MVP 완주**.
- 사용자 경험 목표: 빌드·실행·테스트 명령이 README와 **일치**하고, 에러 시 **stderr**에 원인을 알 수 있다.
- 기술 목표: **파싱 결과(AST 또는 중간 구조)** 와 **실행(Executor)** · **저장(Storage)** 의 책임 분리.

## 5) MVP 범위

이번 프로젝트에서 반드시 구현할 것:

- 텍스트 **SQL 파일**을 인자로 받는 **CLI**
- **INSERT INTO … VALUES (…)** 파싱 및 **테이블 파일에 행 추가**
- **SELECT … FROM …** 파싱 및 **테이블 파일에서 읽어 stdout 출력**
- **테이블당 하나의 파일**(저장 포맷은 `docs/02-architecture.md` 기준 **CSV**)
- **단위 테스트** + **기능(통합) 테스트**(SQL 픽스처 기준)
- **CREATE TABLE 미구현** — 스키마·테이블 파일은 **사전에 존재**한다고 가정

기능별 상세:


| 기능              | 설명                                        | 우선순위   | 포함 여부 |
| --------------- | ----------------------------------------- | ------ | ----- |
| CLI SQL 파일 실행   | `sql_processor <path.sql>` 형태로 스크립트 실행    | High   | Yes   |
| Lexer/Tokenizer | 식별자, 리터럴, 기호, 키워드 분리                      | High   | Yes   |
| INSERT 파서·실행    | 단일 행 VALUES, CSV 한 줄 append               | High   | Yes   |
| SELECT 파서·실행    | `SELECT `* 또는 `SELECT col1, col2` (컬럼 목록) | High   | Yes   |
| CSV 저장소         | 테이블명→파일 경로, 헤더+데이터 행 규칙                   | High   | Yes   |
| 에러 처리           | 구문 오류·IO 오류 시 비제로 종료 코드 + stderr          | Medium | Yes   |


### Stretch (차별화·여유 시)


| 기능               | 설명                         | 우선순위 | 포함 여부    |
| ---------------- | -------------------------- | ---- | -------- |
| WHERE 절          | 단순 조건(예: `col = value`) 필터 | Low  | Optional |
| ORDER BY / LIMIT | 결과 정렬·개수 제한                | Low  | Optional |
| 트랜잭션·동시성         | 없음                         | —    | No       |


## 6) 비MVP 범위

의도적으로 제외:

- **CREATE / DROP / ALTER TABLE**
- **JOIN**, 서브쿼리, 집계(GROUP BY), HAVING
- 인덱스, 뷰, 권한, 네트워크 프로토콜
- 완전한 SQL 표준 준수

## 7) 핵심 사용자 흐름

### Flow A. INSERT 후 SELECT로 확인

1. 팀은 `data/users.csv` 등 **스키마에 맞는 CSV 파일**(헤더 포함)을 준비한다.
2. `sample.sql`에 `INSERT INTO users VALUES (...);` 와 `SELECT * FROM users;` 를 적는다.
3. 사용자는 빌드 산출물(경로는 `README.md` Quick Start 참고)로 `sql_processor sample.sql` 을 실행한다.
4. 시스템은 문장을 순서대로 파싱·실행하고, INSERT는 파일 끝에 행을 추가하고, SELECT는 **표 형태에 가깝게** stdout에 출력한다.
5. 결과적으로 터미널에서 **데이터 반영 여부**를 확인한다.

### Flow B. 잘못된 SQL

1. 사용자는 문법 오류가 있는 `.sql` 파일을 실행한다.
2. 시스템은 **stderr**에 짧은 메시지를 출력하고 **비제로 종료 코드**로 끝낸다.
3. (선택) 테스트는 기대 stderr/exit code를 검증한다.

## 8) 성공 기준

- 데모 시나리오(README 기준) **1회 이상 완주**
- **INSERT / SELECT** 가 파일과 stdout에 대해 문서(`docs/03-api-reference.md`)와 **동일한 계약**으로 동작
- **ctest** 또는 동등한 자동화로 **단위·통합 테스트** 통과
- README와 실제 CLI 인자·경로·명령이 **크게 어긋나지 않음**
- 발표에서 **파싱 → 실행 → 저장** 중심 로직을 **설명 가능**

## 9) 오픈 질문

- Stretch로 **WHERE** 까지 갈지, MVP만으로 마감할지(일정 우선).
- Windows 전용 vs 크로스플랫폼 빌드(CMake 타겟 경로 문서화).
- SELECT 출력 포맷: TSV / 고정폭 / CSV 복제 중 무엇을 기본으로 할지(`docs/03-api-reference.md`에서 확정).
