# Project Docs Starter

이 폴더는 새 프로젝트를 시작할 때 그대로 복사해서 쓰는 문서 스타터 킷입니다.

권장 사용 방식:

1. (이 저장소는 이미 템플릿을 기반으로 **SQL 처리기** 문서가 채워져 있습니다.) 다른 프로젝트에 쓸 때는 이 폴더 내용을 새 루트로 복사합니다.
2. `AGENTS.md`를 먼저 열어 Codex 작업 규칙과 명령어를 프로젝트에 맞게 채웁니다.
3. `docs/01-product-planning.md`부터 순서대로 채웁니다.
4. MVP 범위가 고정되면 `docs/02-architecture.md`, `docs/03-api-reference.md`를 채웁니다.
5. 작업 단위가 보이기 시작하면 `docs/04-development-guide.md`에서 브랜치/테스트 규칙을 정합니다.
6. 마지막에 `README.template.md`를 `README.md`로 옮기고 발표용으로 다듬습니다(본 repo는 이미 `README.md` 존재).
7. 발표 전에는 README에 `데모 시나리오`, `주요 화면`, `트러블슈팅 / 배운 점`을 꼭 채웁니다.

왜 이런 순서가 좋은가:

- README를 먼저 너무 길게 만들지 않게 도와줍니다.
- 기획, 설계, 구현 계획이 한 문서에 섞이지 않게 해줍니다.
- 프로젝트가 커져도 문서가 망가지지 않습니다.
- 팀원들이 Codex를 써도 같은 규칙을 따라가게 해줍니다.
- 마지막에 README만 정리해도 발표 흐름을 바로 만들 수 있습니다.

추천 작성 순서:

1. 문제 정의 / 타겟 사용자
2. MVP 범위 / 제외 범위
3. 핵심 사용자 흐름
4. 기술 스택 선택
5. 아키텍처와 데이터 모델
6. API 규칙과 엔드포인트
7. 브랜치 계획 / 테스트 계획
8. README 발표 요약본 정리

폴더 구조:

```text
project-root/
├─ AGENTS.md
├─ README.md
└─ docs/
   ├─ 01-product-planning.md
   ├─ 02-architecture.md
   ├─ 03-api-reference.md
   └─ 04-development-guide.md
```

실전 팁:

- README는 "처음 보는 사람 + 발표 청중"이 읽는 문서로 유지합니다.
- 자세한 설계/진행 기록은 `docs/`로 보냅니다.
- Codex에게는 항상 "어떤 문서를 기준으로 작업할지"를 같이 말합니다.
- 기능 PR에는 코드 + 테스트 + 문서 변경을 같이 넣습니다.
- 시퀀스 다이어그램은 모든 기능에 그리지 말고, 로그인/결제/예약처럼 충돌 가능성이 큰 흐름만 우선 그립니다.
- 발표 직전에는 README에 화면 캡처와 데모 시나리오를 넣습니다.
