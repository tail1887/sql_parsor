## 단계 1 — 프로젝트 부트스트랩

**하는 일**: 디렉터리(`include/`, `src/`, `tests/`, `data/`), `CMakeLists.txt`, `main` 골격까지 **빌드·실행·테스트가 도는 최소 뼈대**를 만든다.

**공부 포인트**

- CMake가 **소스 트리**와 **빌드 트리**(`build/`)를 어떻게 나누는지, `add_executable`·`target_include_directories`가 무슨 역할인지.
- C11 플래그가 켜졌을 때 **표준 C**와 구현 확장의 차이(문서상 `CMAKE_C_EXTENSIONS OFF` 의도).
- `enable_testing` / `add_test` / `ctest`로 **회귀 검증 루프**를 만드는 흐름.

**스스로 점검**

- `cmake --build` 후 실행 파일이 생기는가? `ctest`를 한 번이라도 돌려볼 수 있는가?

## 단계 2 — Lexer (토크나이저)

**하는 일**: UTF-8 SQL 텍스트를 앞에서부터 읽어 **토큰 스트림**으로 잘라 낸다. 키워드/식별자/리터럴/구두점/EOF 등 `03-api-reference.md`에 맞는 종류를 정한다.

**공부 포인트**

- **왜 문자 단위가 아니라 토큰인가**: 파서가 다루는 최소 단위를 lexer에서 확정한다는 뜻.
- **최장 일치(longest match)**: `INSERT`를 한 토큰으로 볼지, 식별자와 어떻게 구분할지.
- **키워드 case-insensitive** 정책을 lexer에서 어떻게 반영할지(정규화 시점).
- 문자열 리터럴은 MVP에서 **작은따옴표** (`'...'`, 내부 `**''`** → `'` 한 글자). 닫는 `'` 판별·미종료 처리 — 여기서 틀리면 파서까지 오류가 전파된다. (CSV 직렬화의 큰따옴표 `""` 규칙은 `csv_storage` 단계 이야기이며 SQL 리터럴과 다름.)
- 공백·줄바꿈·(있다면) 주석 — **토큰으로보내지 않고 스킵**하는 규칙을 한곳에 모을 것.

**스스로 점검**

- 짧은 SQL 조각을 넣었을 때 토큰 나열이 **손으로 기대한 순서**와 같은가?
- 잘못된 따옴표·끝나지 않은 문자열에서 **어디서 실패**시킬지 정했는가?

---

## 단계 3 — INSERT 파서 + AST 기초

**하는 일**: 토큰을 소비하면서 `INSERT INTO … VALUES (…);` 만 **구문 트리(AST)** 로 만든다. `Stmt` 종류 중 INSERT 분기와, 테이블명·값 리스트를 담는 필드를 설계한다.

**공부 포인트**

- **문법을 규칙으로 쓰고**, 그 규칙대로 **함수를 나누는** 방식(재귀 하강이 흔함): `insert_statement`, `value_list` 등.
- **lookahead**: 다음 토큰이 `INTO`인지 `VALUES`인지에 따라 분기하는 패턴.
- AST는 **실행기가 필요로 하는 정보만** 담는다 — 텍스트 그대로 vs 정규화된 값(숫자, 문자열, NULL) 구분.
- 메모리: 문자열을 AST에 **복사**할지 포인터만 둘지, 파싱 실패 시 **할당 해제** 순서.

**스스로 점검**

- 문법에 안 맞는 INSERT 여러 개를 넣었을 때 **일관된 에러**가 나는가?
- 성공 시 executor가 읽기만 하면 되도록 **필드가 충분한가**?

---

## 단계 4 — SELECT 파서

**하는 일**: `SELECT * FROM <table>;` (및 문서에 정한 컬럼 목록 형태)를 같은 AST 체계에 **SELECT 분기**로 추가한다.

**공부 포인트**

- INSERT 때 만든 **토큰 소비·에러 처리 패턴**을 재사용하는 법.
- `*` vs 명시적 컬럼 리스트를 AST에서 **어떻게 표현**할지(플래그, 리스트 길이 0 등).
- 키워드 순서가 INSERT와 다르므로 **혼동 없이 규칙 추가**하기.

**스스로 점검**

- 지원 문법 밖 SELECT는 parser 단계에서 걸리는가?

---

## 단계 5 — CSV Storage

**하는 일**: 논리 테이블명 → `data/<table>.csv` 경로, 헤더 읽기, 행 append, (SELECT용) 읽기. 필드 구분·따옴표 이스케이프는 `03-api-reference.md`·`02-architecture.md`와 동일하게.

**공부 포인트**

- **스트림 vs 한 줄 버퍼**: `fgets` 등으로 한 행씩 다룰 때 버퍼 크기·개행 처리.
- RFC 4180 스타일 **콤마/따옴표**가 섞인 필드를 파싱할 때 상태(따옴표 안/밖).
- **INSERT 시 컬럼 수 = 헤더 컬럼 수** 검증을 어디서 할지(storage vs executor — 책임 경계).
- 파일 없음·I/O 실패 시 **실행 오류**로 올릴 정보(errno, 짧은 메시지).

**스스로 점검**

- 손으로 만든 CSV로 “콤마 포함 문자열”“`""` 이스케이프” 케이스를 읽고 쓸 수 있는가?

---

## 단계 6 — Executor

**하는 일**: AST 종류에 따라 Storage를 호출한다. INSERT → `append_row`, SELECT → 읽은 뒤 **stdout 포맷**(`03-api-reference.md`).

**공부 포인트**

- **Parser/Storage와의 경계**: executor는 “무엇을 할지”, storage는 “파일에 어떻게” — 조건 분기는 최소화.
- SELECT 출력: 헤더/행 구분, 구분자, 마지막 줄바꿈 등 **문서와 바이트 단위로 맞출지** 정하기.
- 한 스크립트에 문장이 여러 개일 때 **중간 실패 시** 종료 코드·이후 문장 처리 정책(문서 따르기).

**스스로 점검**

- AST만 보고 동작을 설명할 수 있는가? (테이블 이름·값이 어디서 왔는지)

---

## 단계 7 — CLI 통합

**하는 일**: 인자 검증, SQL 파일 전체 읽기, **세미콜론으로 문장 분리** 후 문장마다 parse → execute, 종료 코드 부여.

**공부 포인트**

- `argc`/`argv`, 파일 열기·크기/버퍼 읽기, 실패 시 stderr 메시지와 **exit code 1 vs 2 vs 3** 매핑(`03-api-reference.md`).
- **문장 분리**를 lexer/parser와 중복하지 않도록: “파일 → 문장 문자열”과 “문장 → 토큰” 역할 분담.
- Windows 경로·UTF-8 파일 vs 콘솔 인코딩 이슈를 발표/문서 수준에서 어떻게 설명할지.

**스스로 점검**

- 잘못된 인자, 깨진 SQL, 없는 테이블 각각이 **문서에 적힌 코드**로 끝나는가?

## 단계 8 — 에러 처리·엣지 케이스

**하는 일**: 빈 파일, 헤더만 있는 CSV, 마지막 세미콜론 유무, 동일 입력에 대한 **안정적** 실패 메시지 등을 문서·테스트와 맞춘다.

**공부 포인트**

- 에러 경로에서 **리소스 해제**(열린 파일, malloc) — 성공 경로만큼 명시적으로.
- 사용자에게 나가는 메시지는 **짧고 일관**되게(디버그 print 남발 줄이기).

**스스로 점검**

- 팀원이 같은 잘못된 입력을 넣어도 같은 종류의 에러로 이해할 수 있는가?

**공부 포인트**
