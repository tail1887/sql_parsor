# WEEK8 Planning — 미니 DBMS API 서버

WEEK8 문서는 `docs/01-product-planning.md`의 기본 제품 방향을 유지하면서,  
API 서버와 멀티스레드 처리 요구를 위한 주차별 확장 계획을 정의합니다.

## 1) 프로젝트 소개

- 프로젝트명: **sql_parsor WEEK8 — Mini DBMS API Server**
- 한 줄 설명: 기존 C 기반 SQL 처리기/CSV 저장소/B+ 트리 인덱스를 재사용해, 외부 클라이언트가 HTTP API로 DB 기능을 호출할 수 있게 하는 API 서버를 구현한다.
- 개발 맥락: 수요 코딩회 1일 몰입 구현 + 목요일 오전 데모 발표

## 2) 문제 정의

- 기존 구현은 CLI 중심이라 외부 시스템 연동이 어렵다.
- 단일 프로세스/순차 실행 모델에서 서버형 동시 처리 모델로 전환이 필요하다.
- AI를 활용해 빠르게 구현하더라도 핵심 로직(요청 처리, 스레드풀, 동시성 안전성)을 팀이 설명 가능해야 한다.

## 3) 타겟 사용자

- 1차 사용자: 프로젝트 팀원(개발/테스트/발표 담당)
- 2차 사용자: 데모를 보는 동료 팀(HTTP 클라이언트 관점에서 API 사용)
- 사용 시나리오: 로컬 환경에서 API 서버 실행 후, HTTP 요청으로 SQL 실행 결과를 검증

## 4) 목표 설정

### 제품 목표

- 외부 클라이언트가 API를 통해 SQL 실행(INSERT/SELECT)을 수행할 수 있다.
- 요청을 Thread Pool 기반으로 병렬 처리한다.
- 기존 SQL 엔진(파서/실행기/B+ 트리)을 API 서버 내부 엔진으로 재사용한다.

### 기술 목표

- API 계약(요청/응답/오류) 고정
- 동시성 안전성(파일 I/O 및 인덱스 접근) 확보
- 테스트 자동화(단위 + API 기능/동시성 시나리오) 확보

### 학습 목표

- 단순 동작 구현을 넘어서, 다음 항목의 원리를 설명 가능하도록 한다.
  - API 레이어와 엔진 레이어의 경계 설계
  - Thread Pool + Queue 동작 원리
  - 락 전략과 동시성 이슈 대응

## 5) 이번 개발 범위

### In Scope (WEEK8 MVP)

- `GET /health` 헬스체크 엔드포인트
- `POST /query` SQL 실행 엔드포인트
- 요청 검증/응답 표준화(JSON envelope)
- 고정 크기 Thread Pool + bounded queue
- 엔진 브리지(`sql text -> parser/executor -> structured result`)
- 기본 오류 매핑(입력 오류/구문 오류/실행 오류/서버 과부하/타임아웃)
- 단위 테스트 + API 기능 테스트 + 동시성 스모크 테스트

### Out of Scope (이번 주차 제외)

- 인증/인가
- 분산 배포, 다중 프로세스 간 합의
- 고급 SQL 확장(JOIN, 복잡한 WHERE, 트랜잭션 완전 지원)
- 완전한 운영 수준 관측 플랫폼 연동

## 6) 핵심 사용자 흐름

### Flow A: 정상 쿼리 실행

1. 사용자가 API 서버를 실행한다.
2. 클라이언트가 `POST /query`로 SQL을 전송한다.
3. 서버가 입력을 검증하고 작업을 큐에 적재한다.
4. Worker가 SQL 엔진을 호출해 결과를 생성한다.
5. 서버가 JSON 응답으로 결과를 반환한다.

### Flow B: 오류 처리

1. 사용자가 잘못된 요청(JSON 형식/필수 필드 누락) 또는 잘못된 SQL을 보낸다.
2. 서버가 규칙에 맞는 오류 코드를 반환한다.
3. 클라이언트는 동일 포맷의 오류 응답으로 원인을 확인한다.

### Flow C: 과부하 상황

1. 짧은 시간에 다수 요청이 유입된다.
2. 큐가 임계치를 넘으면 즉시 거절(백프레셔) 또는 타임아웃 정책을 적용한다.
3. 서버는 중단되지 않고 보호 정책에 따라 응답한다.

## 7) 성공 기준

- 기능 기준
  - `GET /health` 정상 응답
  - `POST /query`로 INSERT/SELECT 성공
  - 입력/구문/실행 오류가 계약된 응답 형태로 반환
- 동시성 기준
  - 동시 요청 처리 중 크래시/데드락/데이터 손상 없음
  - queue full/timeout 정책이 문서대로 동작
- 품질 기준
  - 핵심 모듈 단위 테스트 통과
  - API 기능 테스트 및 동시성 스모크 테스트 통과
- 발표 기준
  - README 기반 4분 데모 완주
  - 핵심 설계 선택 이유와 트레이드오프 설명 가능
