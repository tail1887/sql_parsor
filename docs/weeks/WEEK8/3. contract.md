# WEEK8 Contract — API 요청/응답/오류 계약

이 문서는 WEEK8 API 서버의 외부 계약을 정의합니다.  
구현 세부는 `docs/weeks/WEEK8/architecture.md`를 따르며, 본 문서는 **클라이언트가 신뢰할 수 있는 인터페이스 규칙**을 고정합니다.

## 1) 계약 범위

- 대상 엔드포인트
  - `GET /health`
  - `POST /query`
- 대상 항목
  - 요청 본문(JSON) 형식
  - 응답 형식(success/error envelope)
  - 오류 코드/HTTP 상태코드 매핑
  - 타임아웃/백프레셔 동작 규칙

## 2) 공통 규칙

- 프로토콜: HTTP/1.1 (로컬 데모 기준)
- 콘텐츠 타입: `application/json; charset=utf-8`
- 인코딩: UTF-8
- 식별자: `requestId`가 없으면 서버가 생성할 수 있음
- 시간 단위: 밀리초(`ms`)
- SQL 실행 단위: **기본 1요청 1문장** 권장

## 3) 엔드포인트 계약

### 3.1 `GET /health`

#### 요청

- body 없음

#### 성공 응답 (`200`)

```json
{
  "ok": true,
  "data": {
    "status": "ok"
  },
  "metadata": {
    "service": "sql_parsor_week8_api"
  }
}
```

### 3.2 `POST /query`

#### 요청 바디

```json
{
  "requestId": "optional-client-id",
  "sql": "SELECT * FROM users;",
  "timeoutMs": 2000
}
```

#### 요청 필드 규칙

- `sql` (required)
  - 타입: string
  - 공백만 있는 문자열 금지
  - 최대 길이: 구현 정책값(예: 16KB)
- `requestId` (optional)
  - 타입: string
  - 없으면 서버 생성
- `timeoutMs` (optional)
  - 타입: integer
  - 최소/최대 범위는 서버 정책으로 clamp 가능

#### 성공 응답 (`200`)

```json
{
  "ok": true,
  "data": {
    "headers": ["id", "name", "age"],
    "rows": [
      ["1", "alice", "20"]
    ],
    "affectedRows": 1
  },
  "metadata": {
    "requestId": "srv-20260422-001",
    "latencyMs": 12
  }
}
```

설명:
- `SELECT`의 경우 `headers`, `rows` 중심
- `INSERT`의 경우 `affectedRows` 중심
- 구현에 따라 사용하지 않는 필드는 빈 배열/0/null 정책을 일관되게 유지

## 4) 공통 응답 Envelope

### 4.1 Success Envelope

```json
{
  "ok": true,
  "data": {},
  "metadata": {}
}
```

### 4.2 Error Envelope

```json
{
  "ok": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "human readable message",
    "retryable": false
  },
  "metadata": {
    "requestId": "srv-20260422-001"
  }
}
```

필수 규칙:
- 실패는 항상 `ok: false`
- 에러는 항상 `error.code` 포함
- 사람용 메시지(`message`)와 기계용 코드(`code`) 분리

## 5) 오류 코드 및 HTTP 상태 매핑

| error.code | 의미 | HTTP |
| --- | --- | --- |
| `INVALID_REQUEST` | JSON 형식/필수 필드/타입 오류 | `400` |
| `UNSUPPORTED_MEDIA_TYPE` | Content-Type 불일치 | `415` |
| `SQL_PARSE_ERROR` | SQL 문법 오류 | `422` |
| `SQL_EXEC_ERROR` | 실행 오류(테이블 없음, 컬럼 불일치, IO 등) | `409` 또는 `500` |
| `QUEUE_FULL` | 큐 포화로 수락 불가 | `503` |
| `REQUEST_TIMEOUT` | deadline 초과 | `504` |
| `INTERNAL_ERROR` | 예상치 못한 서버 오류 | `500` |

정책 메모:
- `SQL_EXEC_ERROR`는 세부 원인에 따라 `4xx/5xx` 분기가 가능하지만, 팀 정책을 고정해 일관성을 유지합니다.

## 6) 타임아웃 및 백프레셔 계약

- Queue Admission
  - 큐가 가득 차면 요청을 즉시 거절하고 `QUEUE_FULL` 반환
- Timeout
  - `timeoutMs`가 있으면 요청별 deadline 적용
  - deadline 초과 시 `REQUEST_TIMEOUT` 반환
- Retry 가이드
  - `QUEUE_FULL`, 일부 `REQUEST_TIMEOUT`은 `retryable=true` 권장
  - `SQL_PARSE_ERROR`는 `retryable=false`

## 7) 엔진 결과와 API 매핑 규칙

기존 CLI 문서(`docs/03-api-reference.md`)의 의미를 API로 변환:

- CLI 사용 오류(`exit 1`) 성격 -> API에서는 `INVALID_REQUEST(400)`
- SQL 구문 오류(`exit 2`) -> `SQL_PARSE_ERROR(422)`
- 실행 오류(`exit 3`) -> `SQL_EXEC_ERROR(409/500)`
- 성공(`exit 0`) -> `ok=true`, `200`

## 8) 검증 시나리오 (계약 테스트 기준)

### 시나리오 A: 정상 SELECT
- 입력: `{"sql":"SELECT * FROM users;"}`
- 기대:
  - HTTP `200`
  - `ok=true`
  - `data.headers`/`data.rows` 존재

### 시나리오 B: 잘못된 JSON
- 입력: `{sql: SELECT * FROM users;}`
- 기대:
  - HTTP `400`
  - `error.code=INVALID_REQUEST`

### 시나리오 C: SQL 문법 오류
- 입력: `{"sql":"SELEC * FROM users;"}`
- 기대:
  - HTTP `422`
  - `error.code=SQL_PARSE_ERROR`

### 시나리오 D: 과부하
- 입력: 동시 burst 요청(큐 용량 초과)
- 기대:
  - 일부 HTTP `503`
  - `error.code=QUEUE_FULL`

## 9) 버전 및 변경 관리
