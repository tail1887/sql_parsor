# WEEK8 발표 질문 모음

## 1. 전체/아키텍처
### Q1. WEEK8에서 이전 주차와 가장 달라진 점은 무엇인가요?
A. 내부 SQL 실행을 넘어 외부 API 호출 경로까지 완성한 점입니다. 즉 엔진 기능 자체보다 "요청-처리-응답"의 제품 흐름을 완성한 것이 핵심입니다.

### Q2. 왜 계약을 먼저 고정했나요?
A. 계약을 먼저 고정하면 구현/테스트 기준선이 생깁니다. 팀원마다 기대값이 달라지는 문제를 줄이고, 후반 재작업 비용을 크게 낮출 수 있습니다.

### Q3. Engine Bridge를 두는 이유는 무엇인가요?
A. API 계층과 엔진 계층의 결합도를 낮추기 위해서입니다. 엔진 내부가 바뀌어도 API 계약을 안정적으로 유지할 수 있습니다.

## 2. 모듈 02 (Thread Pool + Queue)
### Q1. 왜 요청당 스레드 생성 대신 고정 스레드풀을 썼나요?
A. 요청당 생성은 직관적이지만 부하가 커지면 생성/파괴 비용과 컨텍스트 스위칭 비용이 커집니다. 고정 스레드풀은 상한 제어와 예측 가능성이 좋아 운영 안정성에 유리합니다.

### Q2. bounded queue를 둔 핵심 이유는?
A. 무제한 대기 대신 통제 가능한 실패로 바꾸기 위해서입니다. 큐 상한이 없으면 지연과 메모리 사용량이 폭증해 장애가 더 커집니다.

### Q3. 503 증가가 꼭 나쁜 신호는 아닌가요?
A. 항상 그렇지는 않습니다. 서버가 감당 가능한 수준을 넘는 입력을 명시적으로 거절해 전체 시스템 생존성을 지킨 결과일 수 있습니다.

## 3. 모듈 06 (timeout/backpressure/cancellation)
### Q1. 503과 504를 왜 나눠야 하나요?
A. 실패 성격이 다르기 때문입니다. 503은 수용 불가, 504는 수용 후 기한 초과라서 클라이언트 재시도 전략이 달라집니다.

### Q2. 동적 timeout은 언제 도입해야 하나요?
A. 고정 timeout이 부하 변동을 못 따라가서 성공률이나 지연이 크게 흔들릴 때 도입합니다. 단, 큐 길이/지연 메트릭 수집이 먼저 안정화되어야 합니다.

### Q3. cancellation은 실제 실행 중단까지 보장하나요?
A. 구현 방식에 따라 다릅니다. cooperative cancellation은 체크포인트에서 반영되므로 "즉시 강제 종료"와는 다릅니다. 발표에서는 이 경계를 명확히 설명해야 합니다.

## 4. 성능 검증/그래프
### Q1. 비교 그래프의 신뢰성을 어떻게 보장하나요?
A. 동일 환경, 동일 부하, 반복 측정(최소 3회), raw 로그 보관을 고정 원칙으로 둡니다.

### Q2. 어떤 지표를 최소로 제시해야 하나요?
A. throughput, p95 latency, 503/504 오류 비율 3개는 최소로 제시하는 것이 좋습니다.

### Q3. 그래프보다 더 중요한 검증 근거는 무엇인가요?
A. 재현 가능한 측정 절차와 원본 데이터입니다. 숫자만 보여주면 해석 싸움이 되지만, 절차와 로그가 있으면 근거 기반 토론이 가능합니다.

## 5. 시연
### Q1. 시연이 실패하면 어떻게 대응하나요?
A. 미리 정한 대체 플랜으로 즉시 전환합니다. 예를 들어 queue full이 재현되지 않으면 queue/worker 설정을 조정해 재현성을 확보한 후 다시 시도합니다.

### Q2. 시연에서 반드시 보여줘야 할 3가지는?
A. 정상 200, 과부하 503, timeout 504입니다. 이 세 가지가 보여야 기능+안정성+정책 일관성을 함께 증명할 수 있습니다.
