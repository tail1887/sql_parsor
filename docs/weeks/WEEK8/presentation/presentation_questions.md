# WEEK8 발표 질문 통합본 (중복 제거 + 상세 답변)

## 1) 전체 아키텍처 / 계약
### Q1. WEEK8에서 이전 주차와 가장 달라진 점은 무엇인가요?
A. 이전 주차는 내부 SQL 처리 기능 완성에 초점이 있었다면, WEEK8은 외부 클라이언트가 HTTP API로 호출하는 실제 서비스 경로를 완성한 점이 가장 큽니다. 즉 기능 단위 구현에서 시스템 단위 통합으로 단계가 올라갔다고 설명하면 됩니다.

### Q2. 왜 구현 전에 API 계약부터 고정했나요?
A. 계약이 먼저 고정되어야 구현/테스트/문서가 같은 기준을 공유합니다. 계약 없이 코드를 먼저 작성하면 필드명·오류코드·상태코드가 각 모듈마다 달라져 통합에서 재작업이 크게 발생합니다.

### Q3. API 계약은 요청/응답 형식만 정하면 충분한가요?
A. 부족합니다. 최소한 필드 스키마, 오류코드 체계, HTTP 상태코드 매핑, retryable 기준, timeout/queue-full 정책까지 함께 정의해야 클라이언트가 예측 가능한 처리 전략을 세울 수 있습니다.

### Q4. 계약 변경이 생기면 가장 먼저 무엇을 확인하나요?
A. 영향 범위를 먼저 봅니다. 문서, 서버 응답 포맷, 테스트 fixture, 클라이언트 파서가 동시에 깨질 수 있기 때문에 계약 변경은 코드 변경보다 먼저 호환성 체크리스트로 통제하는 것이 안전합니다.

### Q5. 에러 코드를 HTTP 상태코드와 분리한 이유는 무엇인가요?
A. HTTP 상태코드는 전송 계층 의미이고, 에러 코드는 도메인 의미입니다. 둘을 분리하면 같은 4xx라도 파싱 오류/실행 오류/입력 오류를 정밀하게 구분해 클라이언트가 재시도나 사용자 메시지를 다르게 처리할 수 있습니다.

### Q6. 에러 코드는 어떤 기준으로 나눴나요?
A. 실패가 발생한 계층 기준으로 나눴습니다. 입력 검증 실패(API), SQL 파싱 실패(Parser), 실행 실패(Engine), 과부하/timeout(Infra)로 분리하면 운영 중 원인 추적이 훨씬 쉬워집니다.

### Q7. 현재 계약이 테스트 가능한 형태인지 어떻게 판단하나요?
A. 정상·입력오류·구문오류·실행오류·과부하·timeout 케이스 각각에 대해 요청 예시와 기대 응답이 명확하면 테스트 가능하다고 봅니다. “무엇이 정답인지”가 문장으로 표현 가능해야 자동화도 가능합니다.

### Q8. 이 계약이 확장에도 견딜 수 있는 구조인가요?
A. 네, envelope 구조(`ok/data/error/metadata`)를 유지하면 필드 확장이 비교적 안전합니다. 핵심은 기존 필드의 의미를 바꾸지 않고 새 필드를 추가하는 backward-compatible 전략을 지키는 것입니다.

### Q9. 이 문서가 실제 구현 품질에 어떻게 기여했나요?
A. 모듈별 구현 순서와 테스트 기대값이 문서 기준으로 고정되어 통합 시 충돌이 줄었습니다. 특히 `/query` 오류 매핑(400/422/409/503/504)이 흔들리지 않아 회귀 테스트 유지가 쉬웠습니다.

- API 계약 문서와 실제 응답이 어긋나는 것을 자동으로 감지하는 방법은 무엇인가요?

## 2) W8-01 tiny HTTP 서버
### Q1. 왜 라이브러리 대신 tiny 서버를 직접 구현했나요?
A. 과제 목적이 네트워크/서버 기본기 학습이기 때문입니다. 직접 구현하면 `socket-bind-listen-accept`와 응답 작성 흐름, 종료 처리의 책임을 정확히 이해할 수 있습니다.

### Q2. `Connection: close`를 선택한 이유는 무엇인가요?
A. 초기 구현에서는 단순성과 디버깅 용이성이 우선입니다. keep-alive를 넣으면 연결 재사용·요청 경계·idle timeout 관리가 추가되어 복잡도가 급증합니다.

### Q3. graceful shutdown을 왜 초기 단계에서 넣었나요?
A. 종료 경로는 운영에서 자주 깨지는 부분이라 초기에 고정해야 이후 확장(스레드풀/큐/watcher)에서도 누수 없이 정리할 수 있습니다.

### Q4. 서버 기동 실패를 어떻게 진단하나요?
A. 단계별로 실패를 분리해 로그를 봅니다. `socket`, `bind`, `listen` 중 어디에서 실패했는지 확인하고, 포트 충돌/권한/주소 바인딩 문제부터 점검합니다.

### Q5. 라우팅 최소 범위를 어디까지 잡는 게 적절한가요?
A. 첫 단계는 `/health`와 핵심 비즈니스 경로(`/query`)만으로 충분합니다. 최소 경로를 안정화한 뒤 확장하는 방식이 실패 비용이 낮습니다.

### Q6. 운영 관점에서 첫 서버에 꼭 있어야 할 요소는 무엇인가요?
A. 헬스체크, 명확한 오류 응답, 종료 처리, 최소 로그가 필수입니다. 이 네 가지가 있어야 장애 재현과 원인 분석이 가능합니다.

## 3) W8-02 Thread Pool / Queue
### Q1. 왜 요청당 스레드 생성 대신 고정 스레드풀을 선택했나요?
A. 요청당 생성은 간단하지만 부하 시 생성/소멸 오버헤드와 컨텍스트 스위칭 비용이 크게 늘어납니다. 고정 풀은 상한 제어가 가능해 지연과 자원 사용을 예측하기 쉽습니다.

### Q2. 스레드풀과 요청당 스레드 생성의 본질적 차이는 무엇인가요?
A. 본질은 “스레드 재사용 여부”입니다. 스레드풀은 생명주기를 길게 가져가 작업만 교체하고, 요청당 생성은 작업마다 스레드 생명주기를 새로 만듭니다.

### Q3. bounded queue를 둔 핵심 이유는 무엇인가요?
A. 무제한 대기는 장애를 늦게 터뜨립니다. bounded queue는 수용 가능한 상한을 정해 실패를 조기에 표면화하고 시스템을 보호합니다.

### Q4. 큐 포화 시 정책은 어떤 기준으로 결정하나요?
A. 목표가 “지연 최소화”인지 “성공률 최대화”인지로 결정합니다. 이번처럼 안정성/예측 가능성을 우선하면 즉시 거절(503)이 합리적입니다.

### Q5. worker 수는 어떤 근거로 정하나요?
A. CPU 코어 수, 요청의 CPU/IO 비중, 큐 대기시간을 함께 봅니다. 초기값은 보수적으로 잡고 측정 결과로 튜닝합니다.

### Q6. shutdown에서 반드시 보장해야 할 동작은 무엇인가요?
A. 신규 요청 중단, 대기 워커 깨우기, 진행 중 작업 정리, 자원 해제 순서 보장입니다. 순서가 깨지면 데드락/FD 누수 위험이 큽니다.

### Q7. 워크스틸링을 당장 채택하지 않은 이유는 무엇인가요?
A. 기준선이 없는 상태에서 고급 스케줄러를 도입하면 개선 효과를 증명하기 어렵습니다. 먼저 단순한 기준 구현을 만들고 실험으로 확장하는 편이 안전합니다.

### Q8. 503 증가가 꼭 나쁜 신호는 아닌가요?
A. 아닙니다. 과부하를 통제 가능한 실패로 전환한 결과일 수 있습니다. 중요한 건 503 자체보다 p95/p99, 시스템 생존성, 복구 시간과 함께 해석하는 것입니다.

## 4) W8-03 Engine Bridge
### Q1. 왜 엔진을 새로 만들지 않고 브리지 계층을 두었나요?
A. 이미 검증된 SQL 처리 자산을 재사용하면서 API 계층과의 결합도를 낮추기 위해서입니다. 브리지는 “결과 번역기” 역할을 하므로 유지보수 경계를 명확히 해줍니다.

### Q2. 브리지 계층의 성공/실패 경계는 어디인가요?
A. 엔진이 반환한 결과를 API 계약으로 변환할 수 있으면 성공, 변환 불가·메모리 실패·내부 실행 실패는 브리지 실패로 봅니다.

### Q3. 브리지에서 stdout/stderr를 구조화하는 이유는 무엇인가요?
A. CLI 출력은 사람 친화적이지만 API는 기계가 읽어야 합니다. 구조화된 필드가 있어야 오류 매핑, 로깅, 자동 대응이 가능합니다.

### Q4. 메모리 소유권(할당/해제)은 어떻게 설계했나요?
A. 생성자와 해제자 책임을 함수 단위로 명확히 분리합니다. 예: `execute_sql`이 생성한 버퍼는 `result_free`가 반드시 해제하도록 계약화합니다.

### Q5. B+트리를 서버에서도 재사용하는 실익은 무엇인가요?
A. 조회 성능 자산을 그대로 활용하고, 기능 중복 구현을 피할 수 있습니다. 재사용은 성능뿐 아니라 회귀 위험을 낮추는 효과도 있습니다.

### Q6. 재사용 설계가 실패하는 대표 원인은 무엇인가요?
A. 모듈 경계가 불명확해 내부 세부사항에 의존할 때 실패합니다. 인터페이스 계약을 먼저 고정하지 않으면 재사용이 오히려 결합도를 높입니다.

## 5) W8-04 `/query` 통합
### Q1. `/query` 엔드포인트를 별도로 둔 이유는 무엇인가요?
A. 상태 점검(`/health`)과 비즈니스 실행(`/query`) 책임을 분리하면 운영/테스트/장애 대응이 단순해집니다.

### Q2. “긴 쿼리에서 연결 점유”는 정확히 어떤 의미인가요?
A. 동기 모델에서 요청이 끝날 때까지 해당 소켓과 worker가 점유된다는 뜻입니다. 동시에 처리 가능한 슬롯이 줄어드는 효과가 있습니다.

### Q3. 프로세스 블로킹과 스레드 블로킹은 어떻게 구분하나요?
A. 멀티스레드에서는 호출한 스레드가 블로킹되고 프로세스 전체가 멈추는 건 아닙니다. 단, 워커 수가 적으면 체감상 전체가 느려질 수 있습니다.

### Q4. 동기 응답 모델의 장단점을 어떻게 설명하나요?
A. 장점은 구현 단순성과 추적 용이성, 단점은 장기 요청이 자원 점유를 늘린다는 점입니다. 그래서 timeout/backpressure 같은 보완 정책이 필수입니다.

### Q5. 입력 검증은 어느 레이어에서 끝내는 게 맞나요?
A. 가능한 API 레이어에서 끝내는 게 맞습니다. 잘못된 요청을 엔진까지 보내지 않으면 오류 분류가 명확해집니다.

### Q6. 오류 매핑을 API 레이어에서 한 이유는 무엇인가요?
A. 엔진 내부 상태를 외부 계약 코드로 통일하는 책임이 API에 있기 때문입니다. 클라이언트 안정성은 API 계약 일관성에서 나옵니다.

### Q7. 동기 모델의 단점은 어떻게 완화할 수 있나요?
A. queue 상한, timeout, cancel token, watcher 같은 보호장치를 넣어 장기 점유 리스크를 줄입니다.

## 6) W8-05 동시성/락
### Q1. 전역 mutex 1개 방식은 정확히 어떤 방식인가요?
A. 모든 테이블/작업 경로를 단일 뮤텍스로 직렬화하는 방식입니다. 구현은 단순하지만 병렬성은 거의 포기합니다.

### Q2. 전역 mutex면 병렬 처리 의미가 없는 것 아닌가요?
A. 완전 무의미하진 않지만 성능 이점이 제한적입니다. 안전성은 빠르게 확보할 수 있어 초기 디버깅에는 유리합니다.

### Q3. 테이블 단위 rwlock과 파일락은 어떻게 다르나요?
A. rwlock은 스레드 단위 동시성 제어, 파일락은 프로세스 간 제어에 강점이 있습니다. 파일락은 플랫폼/디버깅 난도가 더 큽니다.

### Q4. 쓰기는 원자성을 지켜야 하니 결국 병렬 처리가 안 되나요?
A. 같은 테이블 동일 자원에 대한 쓰기는 직렬화가 필요합니다. 대신 다른 테이블/읽기 경로에서 병렬성을 확보할 수 있습니다.

### Q5. 쓰기 경로에서 캐시는 사용할 수 없나요?
A. 사용할 수는 있지만 정합성 규칙이 더 복잡해집니다. 캐시 일관성 정책 없이 쓰기 캐시를 넣으면 오히려 오류 위험이 증가합니다.

### Q6. race condition은 어떻게 재현하고 검증하나요?
A. 동시 INSERT/SELECT를 반복 실행하고 결과 정합성을 자동 검증합니다. 한 번 성공보다 반복 재현 가능성이 중요합니다.

### Q7. 데드락을 예방하는 최소 규칙은 무엇인가요?
A. 락 획득 순서를 전역 규칙으로 고정하고, 실패 경로에서도 항상 동일 순서로 해제합니다.

### Q8. 실패 경로에서 정합성은 어떻게 보장하나요?
A. 부분 성공 상태를 허용하지 않도록 최소 원자 단위를 정합니다. 실패 시 롤백 또는 불일치 감지·차단 정책이 필요합니다.

### Q9. 락 granularity는 어떤 기준으로 결정했나요?
A. 안전성/병렬성/복잡도의 균형으로 결정합니다. 이번에는 테이블 단위로 타협해 구현 난도와 성능을 맞췄습니다.

### Q10. INSERT 경로에서 같은 임계영역으로 묶어야 할 단위는 무엇인가요?
A. 데이터 append와 인덱스 갱신은 같은 임계영역으로 묶어야 합니다. 둘 중 하나만 성공하면 정합성이 깨집니다.

- 읽기 우선/쓰기 우선 정책은 어떤 상황에서 바꿔야 하나요?

## 7) W8-06 timeout/backpressure/cancellation
### Q1. 스레드풀이 있는데 왜 backpressure가 필요한가요?
A. 스레드풀은 실행 상한만 제한합니다. 입력 폭주를 제한하지 않으면 큐 적체로 지연이 폭증합니다.

### Q2. 503과 504를 왜 나누고, 어떻게 구분하나요?
A. 503은 수용 자체 불가, 504는 수용 후 기한 초과입니다. 실패 지점이 달라 재시도 전략도 달라져야 합니다.

### Q3. retryable 필드는 왜 필요한가요?
A. 클라이언트가 즉시 재시도 여부를 결정할 수 있어 전체 시스템의 회복 속도가 좋아집니다.

### Q4. timeout 기준값은 어떻게 정하나요?
A. 평균이 아니라 p95/p99 지연과 사용자 SLA를 기준으로 시작값을 정하고, 운영 지표로 조정합니다.

### Q5. backpressure 정책은 어떤 기준으로 선택하나요?
A. 서비스 목표(성공률 우선 vs 지연 우선)와 운영 단순성 요구를 함께 봅니다. 발표에서는 정책 선택 근거를 숫자로 제시하는 것이 중요합니다.

### Q6. 보호 정책 도입 효과를 어떻게 입증하나요?
A. 도입 전후 throughput, p95/p99, 503/504 비율, 성공률을 동일 조건에서 비교합니다.

### Q7. timeout에서 가장 중요한 설계 포인트는 무엇인가요?
A. timeout 의미를 명확히 정의하는 것입니다. 응답만 timeout인지 실제 실행 중단까지 포함하는지 경계가 분명해야 합니다.

### Q8. 동적 timeout은 언제 도입해야 하나요?
A. 고정 timeout이 부하 변화에 취약해 지연/성공률이 흔들릴 때 도입합니다. 단, 메트릭 관측이 충분히 성숙한 이후가 안전합니다.

### Q9. cancellation은 실제 실행 중단까지 보장하나요?
A. cooperative cancellation은 체크포인트 기반이라 즉시 강제 종료와 다릅니다. 엔진 루프 설계에 따라 반응 시간 차이가 납니다.

#### 추가 보충 질문
- timeout 값을 사용자별/SLA별로 다르게 주려면 구조를 어떻게 바꿔야 하나요?
- watcher thread가 많아질 때 관리 비용을 줄이는 자료구조는 무엇인가요?
- retryable=true를 잘못 설정했을 때 생기는 운영 리스크는 무엇인가요?

## 8) 테스트/검증 (W8-07~W8-08)
### Q1. 왜 단계별 단위 테스트를 강하게 권장하나요?
A. 결함을 가장 가까운 모듈에서 빨리 잡아야 통합 디버깅 비용이 폭증하지 않습니다.

### Q2. 단위 테스트와 통합/E2E 테스트의 역할 차이는 무엇인가요?
A. 단위는 모듈 계약 검증, 통합/E2E는 모듈 간 연결과 사용자 경로 검증입니다. 둘 중 하나만으로는 품질 설명이 불충분합니다.

### Q3. 단위 테스트가 모두 통과해도 E2E가 필요한 이유는 무엇인가요?
A. 실제 장애는 경계면(라우팅-파싱-엔진-응답 변환)에서 발생합니다. E2E는 이 경계 리스크를 잡는 마지막 안전망입니다.

### Q4. 좋은 단위 테스트의 조건은 무엇인가요?
A. 입력과 기대 결과가 명확하고, 실패 시 원인 추적이 쉬워야 합니다. 테스트 이름도 시나리오를 설명해야 합니다.

### Q5. flaky 테스트를 발견하면 어떻게 대응하나요?
A. 재시도로 덮지 말고 시간 의존성·공유 상태·경합 조건을 제거합니다. 원인 고정 후 안정화가 원칙입니다.

### Q6. 테스트 범위는 어떤 순서로 확장하나요?
A. smoke -> 기능 -> 동시성/성능 순서가 일반적으로 효율적입니다. 원인 분리가 쉬운 순서입니다.

### Q7. AI 생성 코드 검증에서 핵심은 무엇인가요?
A. 빠른 생성 속도를 테스트 밀도로 보완하는 것입니다. 특히 실패 케이스를 먼저 고정해야 품질 리스크가 줄어듭니다.

### Q8. 동시성 테스트의 성공 기준은 무엇인가요?
A. 단순 성공률이 아니라 정합성 유지, 오류 분포 일관성, 지연 임계치 충족을 함께 봐야 합니다.

### Q9. 성능 수치를 해석할 때 주의점은 무엇인가요?
A. 조건 없는 숫자는 의미가 약합니다. 요청 믹스, 동시성, 환경 스펙과 함께 해석해야 합니다.

### Q10. 발표용 지표는 어떻게 선택해야 하나요?
A. 재현 가능성과 설명 가능성을 기준으로 선택합니다. throughput, p95/p99, 503/504, 성공률이 기본 축입니다.

## 9) 발표/문서화 (W8-09)
### Q1. README 중심 발표에서 가장 중요한 것은 무엇인가요?
A. 재현성입니다. 다른 사람이 문서만 보고 동일 결과를 낼 수 있어야 신뢰를 얻습니다.

### Q2. 왜 “무엇을 했나”보다 “왜 그렇게 선택했나”를 강조하나요?
A. 구현 사실은 결과이고, 의사결정 근거가 역량을 보여줍니다. 대안 비교와 트레이드오프가 있어야 설득력이 생깁니다.

### Q3. 발표 흐름은 어떻게 구성하면 설득력이 높나요?
A. 문제 -> 선택 -> 근거 -> 검증 -> 한계/개선 순서가 가장 이해가 쉽습니다.

### Q4. 데모 실패에 대비한 실무적 준비는 무엇인가요?
A. 플랜B를 문서화해야 합니다. 대체 명령, 사전 로그 캡처, fallback 시나리오를 준비하면 발표 리스크가 크게 줄어듭니다.

### Q5. 포트폴리오로 재사용하려면 무엇이 추가로 필요할까요?
A. 코드만으로는 부족합니다. 실행 가이드, 성능 근거, 설계 의사결정 기록이 함께 있어야 완성도 있게 전달됩니다.

### Q6. 발표에서 기술 선택을 어떻게 설명해야 하나요?
A. “선택 이유 + 포기한 것 + 보완책” 3단으로 설명하면 됩니다. 이 구조가 가장 실무적으로 들립니다.

### Q7. Q&A 대비에서 가장 중요한 준비는 무엇인가요?
A. 암기보다 근거 준비입니다. 수치, 로그, 실패 사례를 준비하면 질문이 바뀌어도 일관되게 답할 수 있습니다.

## 10) 전체 개념 관통 질문 (핵심)
### Q1. 이 시스템의 핵심 설계 철학을 한 문장으로 말하면 무엇인가요?
A. "예측 가능한 실패를 통해 시스템 전체 안정성을 지키고, 그 근거를 측정 가능한 지표로 증명한다"입니다.

### Q2. WEEK8 구현에서 가장 중요한 트레이드오프 한 가지는 무엇이었나요?
A. 단기 성능 최댓값보다 운영 안정성과 설명 가능성을 우선한 점입니다. 요청당 스레드 생성, 고급 스케줄러 같은 선택지를 비교했지만, 기준선 구현과 검증 가능성을 우선했습니다.

### Q3. 이 프로젝트가 실무 시스템으로 확장될 때 가장 먼저 바꿔야 할 부분은 무엇인가요?
A. 관측성(메트릭/트레이스/로그)과 정책 외부화입니다. timeout/backpressure를 코드 상수로 두지 않고 런타임 설정과 모니터링 기반 제어로 바꾸는 것이 첫 단계입니다.

### Q4. 기술 선택의 정답을 어떻게 판단했나요?
A. 구현 난이도, 안정성, 성능, 검증 가능성을 같은 테이블에서 비교하고, 실제 측정값으로 의사결정을 닫았습니다. 즉 “주장”이 아니라 “실험 결과”를 정답 근거로 삼았습니다.

### Q5. 발표에서 가장 전달하고 싶은 한 가지 메시지는 무엇인가요?
A. 코드를 많이 쓴 것보다, 설계 선택-대안 비교-검증 근거를 끝까지 일관되게 연결한 점이 이 프로젝트의 핵심 성과라는 것입니다.

### Q6. 프록시 서버와 이번 WEEK8 프로젝트는 어떻게 다른가요?
A. 핵심 차이는 "전달 중심 계층"인지 "실행 중심 계층"인지입니다. 프록시(특히 reverse proxy)는 요청을 백엔드로 전달하면서 TLS 종료, 라우팅, 기본 보안 정책, 캐시, 레이트리밋 같은 공통 관문 기능을 담당합니다. 반면 WEEK8 서버는 `/query`를 받아 내부 SQL 엔진을 실제로 실행하고, 결과를 계약된 API 응답으로 변환하는 애플리케이션 실행 계층입니다. 즉 프록시는 요청을 "어디로 보낼지"를 책임지고, WEEK8은 요청을 "어떻게 처리할지"를 책임집니다. 실무에서는 둘이 대체 관계가 아니라 보완 관계이며, 일반적으로 `Client -> Reverse Proxy/WAF -> WEEK8 API -> SQL Engine` 순서로 함께 배치합니다.
