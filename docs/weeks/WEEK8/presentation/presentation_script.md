# WEEK8 발표 대본 (4분용)

## 1. 소개
안녕하세요. 1조 발표를 맡은 여서진입니다.

이번 주 과제는 **미니 DBMS API 서버**를 구현하는 것이었습니다.  
저희는 이전 차수에서 구현한 SQL 처리기와 CSV 저장소, 그리고 B+ 트리 인덱스를 그대로 활용해서, 외부 클라이언트가 HTTP API를 통해 DBMS 기능을 사용할 수 있도록 서버를 구현했습니다.

## 2. 정의
이번 발표에서 설명할 핵심은 세 가지입니다.

첫째, **멀티 스레드 동시성 이슈를 어떻게 다뤘는가**입니다.  
둘째, **내부 DB 엔진과 외부 API 서버 사이 연결을 어떻게 설계했는가**입니다.  
셋째, **API 서버 아키텍처를 어떤 구조로 구성했는가**입니다.

그리고 품질 측면에서는  
단위 테스트를 통한 함수 검증,  
API 서버 기능 테스트,  
그리고 엣지 케이스 검증까지 포함해서 완성도를 높이는 것을 목표로 했습니다.

## 3. 아키텍쳐 및 흐름
전체 구조는 다음과 같이 볼 수 있습니다.
요청 하나의 흐름을 순서대로 보면 이렇습니다.

클라이언트가 `POST /query` 요청을 apiserver에게 먼저 보냅니다.  
API 서버가 요청을 받고 JSON 형식과 SQL 문자열을 확인합니다.  
검증이 끝난 후 요청은 Job Queue에 들어갑니다.  
worker thread 하나가 큐에서 그 요청을 꺼내 Engine Bridge로 전달합니다.  
Engine Bridge는 API 요청을 기존 SQL 처리기가 이해할 수 있는 형식으로 변환한 뒤, 이를 SQL 처리기에 전달합니다.
그러면 기존 엔진이 parser, executor, CSV storage, B+ 트리 인덱스를 이용해 실제 SQL을 실행합니다.  
실행 결과는 다시 bridge를 통해 JSON 응답으로 정리되고, 최종적으로 클라이언트에게 반환되게 됩니다.


## 4. 모듈별 방식과 성능 테스트
이제 발표할 때 사용할 비교 문서를 기준으로 설명드리겠습니다.

### 4-1. 02 — Thread Pool 및 Job Queue
여기서는 두 방식을 비교했습니다.

- 방식 A: **고정 크기 스레드 풀 + bounded queue**
- 방식 B: **요청당 스레드 생성**

방식 A의 장점은 메모리 상한을 통제할 수 있고, 백프레셔 정책이 명확하다는 점입니다.  
단점은 피크 상황에서 일부 요청을 거절해야 할 수 있다는 점입니다.

반면 방식 B는 구현은 직관적이지만, 스레드 생성 비용과 컨텍스트 스위칭 비용이 커지고, 폭주 상황에서 불안정해질 수 있습니다.

그래서 저희는 과제 요구사항인 **Thread Pool 기반 병렬 처리**를 만족시키기 위해 방식 A를 실제 구현 구조로 채택했고,  
그래프 비교에서는 두 방식의 차이를 실측으로 확인했습니다.

### 4-2. 06 — 타임아웃, 백프레셔, 요청 취소 정책
여기서는 두 정책을 비교했습니다.

- 방식 A: **고정 timeout + 큐 full 즉시 거절**
- 방식 B: **동적 timeout(큐 길이 기반)**

방식 A의 장점은 구현이 단순하고, 시스템 보호 기준이 분명하다는 점입니다.  
단점은 요청 성공률이 떨어질 수 있다는 점입니다.

반면 방식 B는 현재 큐 길이에 따라 더 적응적으로 반응할 수 있지만, 정책이 복잡하고 예측성이 낮아질 수 있습니다.
우리는 B 방식을 채택함.

## 시연
이제 수요코딩회 요구사항 시연을 보여드리겠습니다.
먼저 00_start_server.command를 실행해서 서버를 띄웁니다.
이 상태에서 /health 요청 결과가 {"ok":true}로 나오면, 서버가 정상적으로 살아 있고 요청을 받을 준비가 되어 있다는 뜻입니다.
그다음 bench_small 파일을 조회하는 01_normal_query.command를 실행합니다.
여기서 HTTP 200과 ok:true가 나오면,
API 요청을 받고, SQL을 실행한 뒤, 결과를 다시 응답으로 반환하는 전체 흐름이 정상적으로 연결되어 있다는 것을 확인할 수 있습니다.
다음은 병렬 처리와 보호 정책을 함께 보는 시연입니다.
02_overload.command를 실행해서 총 5000개의 요청을 보내고, 이 중 최대 512개가 동시에 들어오도록 부하를 줍니다.
이 상황에서 확인할 포인트는 세 가지입니다.
첫째, 요청이 많이 몰려도 서버가 멈추거나 종료되지 않아야 합니다.
둘째, 처리 가능한 요청은 정상적으로 200 응답으로 처리되어야 합니다.
셋째, 감당할 수 없는 요청에 대해서는 정책에 따라 명확하게 실패해야 합니다.
실제 결과를 보면,
큐가 가득 차서 더 이상 수용할 수 없는 요청은 503으로 즉시 거절되고,
수용은 했지만 처리 시간이 제한 시간을 넘긴 요청은 504로 종료됩니다.
즉, 이 시연에서 보여드리고 싶은 핵심은
요청이 몰려도 서버가 무너지지 않고, 처리 가능한 것은 처리하고, 불가능한 것은 정해진 정책대로 거절하거나 종료한다는 점입니다.
마지막으로 kill 명령으로 서버를 정상 종료합니다.
