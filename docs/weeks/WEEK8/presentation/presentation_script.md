# WEEK8 발표 대본 (4분용)

## 1. 소개
안녕하세요. 1조 발표를 맡은 여서진입니다.
저희는 이전 차수에서 구현한 SQL 처리기와 CSV 저장소, 그리고 B+ 트리 인덱스를 그대로 활용해서, 외부 클라이언트가 HTTP API를 통해 DBMS 기능을 사용할 수 있도록 미니 API 서버를 구현했습니다.

## 2. 정의
이번 발표에서 설명할 핵심은 세 가지입니다.

첫째, **멀티 스레드 동시성 이슈를 어떻게 다뤘는가  
둘째, **내부 DB 엔진과 외부 API 서버 사이 연결을 어떻게 설계했는가 
셋째, **API 서버 아키텍처를 어떤 구조로 구성했는가

## 3. 아키텍쳐 및 흐름
전체 구조는 다음과 같습니다.
요청 하나의 흐름을 설명드리겠습니다.

클라이언트가 `POST /query` 요청을 API 서버에게 먼저 보냅니다.  
API 서버가 요청을 받고 JSON 형식과 SQL 문자열을 확인합니다.  
검증이 끝난 후 요청은 Job Queue에 들어갑니다.  
worker thread 하나가 큐에서 그 요청을 꺼내 Engine Bridge로 전달합니다.  
Engine Bridge는 API 요청을 기존 SQL 처리기가 이해할 수 있는 형식으로 변환한 뒤, 이를 SQL 처리기에 전달합니다.  

여기서 동시성 제어를 위해 read/write lock을 사용했습니다.  
그러면 기존 엔진이 실제 SQL을 실행합니다.
실행 결과는 다시 bridge를 통해 JSON 응답으로 정리되고, 최종적으로 클라이언트에게 반환되게 됩니다.

## 4. 모듈별 방식과 성능 테스트
4-1. 02 — Thread Pool 및 Job Queue
여기서는 두 방식을 비교했습니다.
•	방식 A는, 고정 크기 스레드 풀 과 bounded queue이고,
•	방식 B는 요청당 스레드 생성 하는 방식입니다.
저희는 서버가 감당할 수 있는 만큼만 처리하고, 넘치는 요청에는 바로 백프레셔를 주기 위해 Thread Pool을 선택했습니다. 이 그래프는 503 비율 그래프인데, 부하가 커질수록 pool 방식이 넘친 요청에 503으로 응답하면서 백프레셔를 주는 것을 보여줍니다.
4-2. 06 — 타임아웃, 백프레셔, 요청 취소 정책
여기서는 두 정책을 비교했습니다.
•	방식 A는 고정 timeout과 큐 full 즉시 거절이고,
•	방식 B는 동적 timeout(큐 길이 기반) 정책입니다.

저희는 요청이 몰릴 때 서버 상태에 맞게 대응하기 위해 dynamic timeout을 선택했습니다. 표를 보면 dynamic timeout은 fixed보다 더 많은 요청을 처리했고, 응답도 더 빨랐으며, timeout 실패도 거의 없었습니다. 즉, 서버가 바쁠수록 모든 요청에 같은 기준을 주는 것보다, 현재 큐 상태를 반영해서 처리하는 방식이 더 효과적이었습니다.

## 시연
이제 수요코딩회 요구사항 시연을 보여드리겠습니다.
파일을 조회하는 normal_query.command를 실행해보겠습니다.
여기서 HTTP 200과 ok:true가 나오면,
API 요청을 받고, SQL을 실행한 뒤, 결과를 다시 응답으로 반환하는 전체 흐름이 정상적으로 연결되어 있다는 것을 확인할 수 있습니다.
다음은 병렬 처리와 보호 정책을 함께 보는 시연입니다.
overload.command를 실행해서 총 5000개의 요청을 보내고, 이 중 최대 512개가 동시에 들어오도록 부하를 줍니다.
실제 결과를 보면,
큐가 가득 차서 더 이상 수용할 수 없는 요청은 503으로 즉시 거절되고,
수용은 했지만 처리 시간이 제한 시간을 넘긴 요청은 504로 종료됩니다.
즉, 과부하 상황에서도 멈추거나 깨지지 않고,
정상 처리와 실패 처리를 모두 정책에 맞게 수행한다는 점을 확인할 수 있습니다.
