# WEEK8 발표 대본 (4분용)

## 1. 소개
안녕하세요. 1조 발표를 맡은 여서진입니다.

이번 주 과제는 **미니 DBMS API 서버**를 구현하는 것이었습니다.  
저희는 이전 차수에서 구현한 SQL 처리기와 CSV 저장소, 그리고 B+ 트리 인덱스를 그대로 활용해서, 외부 클라이언트가 HTTP API를 통해 DBMS 기능을 사용할 수 있도록 서버를 C 언어로 구현했습니다.

즉, 이번 주 구현의 핵심은 새로운 DB 엔진을 다시 만드는 것이 아니라,  
기존 엔진을 재사용하면서 그 앞에 API 서버와 병렬 처리 구조를 붙여 실제 요청을 받을 수 있는 시스템으로 확장한 것입니다.

## 2. 정의
이번 발표에서 설명할 핵심은 세 가지입니다.

첫째, **멀티 스레드 동시성 이슈를 어떻게 다뤘는가**입니다.  
둘째, **내부 DB 엔진과 외부 API 서버 사이 연결을 어떻게 설계했는가**입니다.  
셋째, **API 서버 아키텍처를 어떤 구조로 구성했는가**입니다.

그리고 품질 측면에서는  
단위 테스트를 통한 함수 검증,  
API 서버 기능 테스트,  
그리고 엣지 케이스 검증까지 포함해서 완성도를 높이는 것을 목표로 했습니다.

## 3. 아키텍쳐 및 흐름
전체 구조는 다음과 같이 볼 수 있습니다.

외부 클라이언트가 `/health` 또는 `/query` 요청을 보냅니다.  
API 서버는 요청을 받고 기본 검증을 수행합니다.  
그다음 요청을 바로 실행하지 않고 Job Queue에 넣습니다.  
Worker Thread는 큐에서 요청을 꺼내 Engine Bridge로 전달합니다.  
Engine Bridge는 SQL 문자열을 기존 SQL 처리기로 넘깁니다.  
기존 SQL 엔진은 parser, executor, CSV storage, B+ 트리 인덱스를 그대로 사용해 결과를 만듭니다.  
마지막으로 결과를 JSON 응답으로 변환해 클라이언트에 반환합니다.

이 구조에서 이번 주에 새로 구현한 부분은  
API 서버, Thread Pool, Queue, timeout과 backpressure 같은 서버 계층이고,  
이전 주차 자산은 SQL 처리기와 B+ 트리 인덱스입니다.

즉, 서버는 요청 처리와 병렬성을 담당하고,  
엔진은 SQL 해석과 실행을 담당하도록 역할을 분리했습니다.

## 4. 모듈별 방식과 성능 테스트
이제 발표할 때 사용할 비교 문서를 기준으로 설명드리겠습니다.

### 4-1. 02 — Thread Pool 및 Job Queue
여기서는 두 방식을 비교했습니다.

- 방식 A: **고정 크기 스레드 풀 + bounded queue**
- 방식 B: **요청당 스레드 생성**

방식 A의 장점은 메모리 상한을 통제할 수 있고, 백프레셔 정책이 명확하다는 점입니다.  
단점은 피크 상황에서 일부 요청을 거절해야 할 수 있다는 점입니다.

반면 방식 B는 구현은 직관적이지만, 스레드 생성 비용과 컨텍스트 스위칭 비용이 커지고, 폭주 상황에서 불안정해질 수 있습니다.

그래서 저희는 과제 요구사항인 **Thread Pool 기반 병렬 처리**를 만족시키기 위해 방식 A를 실제 구현 구조로 채택했고,  
그래프 비교에서는 두 방식의 차이를 실측으로 확인했습니다.

발표에서는 이 표를 통해  
“왜 thread pool을 선택했는가”를 설명하겠습니다.

### 4-2. 06 — 타임아웃, 백프레셔, 요청 취소 정책
여기서는 두 정책을 비교했습니다.

- 방식 A: **고정 timeout + 큐 full 즉시 거절**
- 방식 B: **동적 timeout(큐 길이 기반)**

방식 A의 장점은 구현이 단순하고, 시스템 보호 기준이 분명하다는 점입니다.  
단점은 요청 성공률이 떨어질 수 있다는 점입니다.

반면 방식 B는 현재 큐 길이에 따라 더 적응적으로 반응할 수 있지만, 정책이 복잡하고 예측성이 낮아질 수 있습니다.

그래서 이 비교에서는  
“어떤 정책이 더 빠른가”보다,  
“과부하 상황에서 어떤 정책이 더 설명 가능하고 제어 가능한 실패를 만드는가”를 중심으로 해석했습니다.

발표에서는 이 표를 통해  
timeout, backpressure, 요청 취소 정책의 설계 의도를 설명하겠습니다.

## 5. 수요코딩회 요구사항 시연
시연은 어렵게 설명하지 않고, 실제로 "잘 동작하는지", "몰리면 어떻게 버티는지"를 눈으로 확인하는 순서로 보여줍니다.

첫째, **기능 시연**입니다.  
먼저 `00_start_server.command`를 실행해서 서버를 띄웁니다.  
여기서 `/health` 결과가 `{"ok":true}`로 나오면, 서버가 기본적으로 살아 있고 요청을 받을 준비가 됐다는 뜻입니다.

그다음 `01_normal_query.command`를 실행합니다.  
`HTTP 200`과 `ok:true`가 나오고, `week8_bench_small` 조회 결과가 정상 출력되면,  
"API 요청 -> SQL 실행 -> 결과 반환" 흐름이 끝까지 제대로 연결됐다는 의미입니다.

둘째, **병렬 처리 + 보호 정책 통합 시연**입니다.  
`02_overload.command`로 `requests=5000`, `concurrency=512` 부하를 한 번에 보냅니다.  
실제 결과는 `503=3542`, `504=1027`, `200=431`이었습니다.

이 결과를 쉬운 말로 풀면 이렇습니다.  
요청이 많이 몰렸는데도 서버가 멈추거나 터지지 않고,  
처리 가능한 요청은 정상 처리하고(`200`),  
너무 몰린 요청은 즉시 거절(`503`)하고,  
처리 시간이 오래 걸린 요청은 제한시간 초과로 중단(`504`)했습니다.

즉, 한 번의 시연으로  
"병렬 처리 중에도 서버가 통제된 규칙대로 안전하게 동작한다"는 점을 보여줍니다.  
이번 데모의 핵심은 성공률 100%가 아니라, 과부하 상황에서도 시스템이 무너지지 않는 구조를 확인하는 것입니다.

그리고 이 시연의 신뢰성을 위해  
`tests/test_week8_engine_bridge.c`,  
`tests/test_week8_api_server.c`,  
`tests/test_week8_locking.c`  
같은 테스트로 함수와 서버 동작을 함께 검증했습니다.

## 6. 마무리
정리하겠습니다.

이번 주 구현의 핵심은  
기존 SQL 처리기와 B+ 트리 인덱스를 재사용하면서,  
그 위에 API 서버와 Thread Pool 구조를 붙여 외부 클라이언트가 사용할 수 있는 미니 DBMS로 확장한 것입니다.

즉, 이번 발표에서는  
요구사항을 어떻게 만족했는지,  
그 과정에서 어떤 구조를 선택했는지,  
그리고 그 선택을 어떻게 테스트와 실측으로 검증했는지를 보여드리겠습니다.

감사합니다.
