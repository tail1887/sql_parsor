# WEEK8 발표 대본 (4분용)

## 0. 시작 멘트
안녕하세요. 1조 발표를 맡은 여서진입니다.

## 1. 프로젝트 소개
이번 주 저희가 한 작업은, 기존에 구현한 SQL 처리기와 CSV 저장소, 그리고 WEEK7의 B+ 트리 인덱스를 그대로 재사용하면서, 외부 클라이언트가 HTTP API로 DB 기능을 호출할 수 있도록 API 서버를 붙이는 것이었습니다.

README 기준으로 보면, 기존 프로젝트가 `입력(SQL) -> 파싱 -> 실행 -> 저장` 구조의 미니 DB였다면, 이번 주에는 그 앞에 API 서버 계층을 추가해 실제 요청-응답 제품 형태로 확장한 것입니다.

쉽게 말하면 흐름은 이렇습니다.  
클라이언트가 요청을 보냅니다.  
서버가 요청을 검증합니다.  
검증된 요청을 병렬 처리 구조에 넣습니다.  
기존 SQL 엔진을 실행합니다.  
그 결과를 JSON 응답으로 바꿔서 돌려줍니다.

## 2. 이번 주 핵심 문제
WEEK8에서 저희가 해결하려 한 문제는 세 가지였습니다.

첫째, **API 계약 문제**입니다.  
`/health`와 `/query`의 입력과 출력 형식이 흔들리면 구현도 테스트도 흔들리기 때문입니다.

둘째, **동시성 문제**입니다.  
요청이 동시에 들어올 때 요청마다 새 스레드를 만드는 방식은 직관적이지만, 부하가 커지면 생성 비용과 컨텍스트 스위칭 비용이 커집니다.

셋째, **보호 정책 문제**입니다.  
받을 수 없는 요청은 적절히 거절해야 하고, 너무 오래 걸리는 요청은 제한해야 서버가 제어 가능한 상태를 유지할 수 있습니다.

그래서 저희는 API 계약, 스레드풀과 큐, timeout과 backpressure를 하나의 묶음으로 설계했습니다.

## 3. 아키텍처와 요청 처리 흐름
이제 구조를 설명드리겠습니다.

저희 구조는 크게 네 계층으로 볼 수 있습니다.

- API Router
- Thread Pool + Queue
- Engine Bridge
- SQL Engine

각 계층의 역할은 명확합니다.

API Router는 `/health`, `/query`를 구분하고, 요청 형식을 검증합니다.  
Thread Pool + Queue는 요청 수락과 실행을 분리해서 병렬 처리를 제어합니다.  
Engine Bridge는 엔진 결과를 API 계약에 맞는 응답 형식으로 정리합니다.  
SQL Engine은 기존에 구현한 parser, executor, CSV storage, B+ 트리 인덱스를 그대로 사용합니다.

실제 요청 흐름은 다음과 같습니다.  
클라이언트가 `/query`를 호출합니다.  
API가 요청을 받고 기본 검증을 합니다.  
그 요청은 큐에 들어갑니다.  
워커 스레드가 큐에서 요청을 꺼내 기존 SQL 엔진을 실행합니다.  
실행 결과는 표준 JSON으로 바뀌고, 최종적으로 HTTP 응답으로 반환됩니다.

이 구조의 장점은 책임이 섞이지 않는다는 점입니다.  
라우팅, 실행, 응답 변환이 분리되어 있기 때문에 문제가 생겨도 어느 계층에서 원인이 났는지 설명하기가 쉽습니다.

## 4. 이번 주 핵심 선택
이번 주에 저희가 강조하고 싶은 선택은 세 가지입니다.

첫째, **고정 크기 스레드풀 + bounded queue**입니다.  
이렇게 하면 서버가 감당할 수 있는 동시 처리량 상한이 분명해집니다.

둘째, **timeout과 backpressure 정책**입니다.  
서버가 바쁠 때는 무한정 기다리게 하지 않고, 거절할 요청과 늦게 끝난 요청을 구분해서 처리하도록 했습니다.

셋째, **오류 의미 분리**입니다.  
`503`은 수용 자체가 불가능한 경우이고,  
`504`는 수용은 했지만 deadline 안에 완료하지 못한 경우입니다.

이 차이를 분명히 해두면, 나중에 과부하 상황을 해석할 때도 실패 원인을 구분해서 설명할 수 있습니다.

## 5. 비교와 실측 요약
발표 시간이 4분이기 때문에 비교 섹션은 숫자를 다 읽기보다 메시지만 전달하겠습니다.
다만 슬라이드에 들어가는 비교 표 자체는 실제 테스트로 검증한 근거 자료이기 때문에 그대로 유지하고, 발표에서는 그 표를 요약해서 설명하는 방식으로 가져가겠습니다.

먼저 스레드 처리 방식 비교에서는, 단순한 `/health` 중심 워크로드에서는 요청당 스레드 생성 방식이 throughput 면에서 조금 더 높게 측정된 구간이 있었습니다.  
하지만 저희는 실제 SQL 실행과 큐 포화 상황까지 고려했을 때, **제어 가능한 병렬성**이 더 중요하다고 판단했습니다.

그리고 timeout 정책 비교에서는, 고부하 구간에서 동적 timeout 방식이 더 안정적인 지연과 낮은 timeout 손실을 보였습니다.  
즉, 이번 실측의 핵심은 “무조건 빠른 방식”을 찾는 것이 아니라, **부하가 걸렸을 때도 어떤 방식이 더 설명 가능한 실패를 만드는가**를 보는 것이었습니다.

그래서 저희는 throughput 하나만 보는 대신,  
throughput, p95 latency, 그리고 503/504 비율을 함께 봤습니다.

## 6. 시연 포인트
시연은 세 가지로 진행합니다.

첫째, `/health`와 `/query`의 정상 응답을 확인합니다.  
둘째, 요청을 몰아넣어 queue full 상황에서 `503`이 나오는지 확인합니다.  
셋째, 오래 걸리는 요청에서 `504`와 timeout 경로가 동작하는지 확인합니다.

시연의 핵심은 “예쁘게 성공하는 것”이 아니라,  
정책이 실제로 재현 가능하게 동작하는 것을 보여주는 것입니다.
