# WEEK8 발표 대본 (실전 구어체)

## 0. 시작 멘트
안녕하세요. 1조 발표를 맡은 여서진입니다.  

## 1. 소개
이번 주는 기존 SQL 처리기/CSV 저장소/B+ 트리 인덱스를 재사용해, 외부 클라이언트가 HTTP API로 DB 기능을 호출할 수 있게 하는 API 서버를 구현하는 것입니다.
WEEK8에서는 그 처리엔진을 API 서버 뒤에 붙여서 실제 요청-응답 제품 형태로 완성했습니다.

쉽게 말하면 다음 5단계입니다.  
요청을 받는다 -> 병렬로 처리한다 -> 엔진 실행한다 -> 결과를 계약된 형식으로 바꾼다 -> 클라이언트에 응답한다.

여기서 중요한 건 단순 성공 케이스가 아니라, 과부하나 지연 상황에서도 서버가 예측 가능한 방식으로 동작해야 한다는 점입니다.  
그래서 이번 발표는 기능 자랑보다 "왜 이 선택을 했고, 어떻게 검증했는지"를 중심으로 말씀드리겠습니다.

## 2. 정의
저희가 정의한 WEEK8 문제는 세 가지입니다.

첫째, **계약 문제**입니다.  
`/health`, `/query`의 요청/응답/오류 포맷이 고정되지 않으면 팀원마다 해석이 달라지고 테스트도 흔들립니다.

둘째, **동시성 문제**입니다.  
요청이 동시에 들어올 때 요청당 스레드를 만들면 순간적으로는 돌아가도 부하가 커질수록 비용이 급격히 늘어납니다.

셋째, **보호 정책 문제**입니다.  
서버가 바쁠 때는 거절할 수 있어야 하고, 너무 오래 걸리는 요청은 끊을 수 있어야 합니다.

그래서 저희는 API 계약, 스레드풀 기반 처리, timeout/backpressure를 하나의 묶음으로 설계했습니다.  
이렇게 해야 실제 운영에서 설명 가능한 시스템이 됩니다.

## 3. 아키텍처 및 요청 처리 흐름
아키텍처는 4계층으로 이해하시면 됩니다.

- API Router: 라우팅과 입력 검증 담당
- Thread Pool + Queue: 요청 수락과 실행 분리
- Engine Bridge: 엔진 결과를 API 계약 형식으로 변환
- SQL Engine: 실제 SQL 파싱/실행

요청 흐름을 구어체로 풀면 이렇습니다.  
"클라이언트가 `/query`를 호출한다 -> API가 기본 검증을 한다 -> 요청을 큐에 넣는다 -> 워커가 큐에서 꺼내 SQL을 실행한다 -> 결과를 표준 JSON으로 만들어 응답한다."

이 구조의 장점은 책임이 분리된다는 점입니다.  
API는 계약을 지키는 데 집중하고, 실행은 워커와 엔진이 담당하므로 문제 발생 시 원인 추적이 쉬워집니다.

## 4. 모듈별 방식 비교 + 성능 테스트 계획

## 4-1. 02 Thread Pool 및 Job Queue
여기는 저희가 가장 먼저 기준선을 잡은 파트입니다.

저희 선택은 **방식 A: 고정 크기 스레드풀 + bounded queue**입니다.  
이걸 고른 이유는 간단합니다. 서버가 감당할 수 있는 동시 처리량 상한을 명확히 만들 수 있기 때문입니다.

대안이었던 **방식 B: 요청당 스레드 생성**은 구현은 쉽지만, 요청이 몰릴 때 생성/파괴 비용과 컨텍스트 스위칭 비용이 크게 늘어납니다.  
결국 순간 피크에서 지연이 폭증하고, 장애가 났을 때 왜 느려졌는지 설명이 어려워집니다.

저희가 발표에서 보여줄 비교 포인트는 세 가지입니다.
- throughput (초당 처리량)
- p95 latency (상위 지연)
- 503 비율 (queue full 거절률)

실제 측정 결과를 말씀드리면,  
normal에서는 per_request가 `11648.91 rps`, pool이 `11447.07 rps`, p95는 각각 `1.49 ms`와 `1.58 ms`였습니다.  
burst에서는 per_request `12240.27 rps`, pool `11410.10 rps`, p95는 `7.10 ms` 대 `8.17 ms`였습니다.  
saturation에서는 per_request `2130.17 rps`, pool `2067.96 rps`, p95는 `12.33 ms` 대 `11.98 ms`였습니다.

이번 벤치가 `/health` 중심 워크로드여서 503 비율은 양쪽 모두 0이었습니다.  
그래서 발표에서는 "현재 실측에서는 per_request가 소폭 유리하게 나왔지만, SQL 실행/큐 포화를 포함하면 결과가 달라질 수 있다"는 한계를 같이 설명하겠습니다.

그리고 이건 꼭 강조하겠습니다.  
503이 나온다고 무조건 나쁜 게 아니라, 서버가 무제한 대기로 무너지는 대신 제어 가능한 실패를 선택했다는 뜻일 수 있습니다.

## 4-2. 06 timeout/backpressure/cancellation
여기는 보호 정책 파트입니다.

정책 비교는 다음 두 가지입니다.
- 방식 A: 고정 timeout + queue full 즉시 거절
- 방식 B: 동적 timeout(큐 길이 기반)

A는 단순하고 예측 가능성이 높습니다.  
B는 상황 적응력이 좋지만 튜닝이 어렵고, 기준이 흔들리면 디버깅 난이도가 올라갑니다.

현재 구현은 `B + M1 + M2` 조합입니다.
- M1: cooperative cancellation (요청별 cancel token 체크)
- M2: watcher thread + deadline registry (만료 시점 트리거)

오류 의미는 분명하게 나눴습니다.
- 503: 수용 자체 불가 (queue full)
- 504: 수용은 했지만 deadline 내 완료 실패

여기서도 비교 지표는 실제 수치로 가져가야 설득력이 생깁니다.
- timeout ratio (504)
- queue reject ratio (503)
- p95/p99 latency

## 5. 비교 섹션 실측 기준 (그래프 생성용)
이번 발표의 비교 그래프는 "추정"이 아니라 "실측" 기반으로 만들겠습니다.

측정 원칙은 4개입니다.
1) 동일 머신, 동일 빌드 옵션  
2) 동일 요청 수, 동일 SQL mix  
3) 3회 이상 반복 후 평균/최댓값 기록  
4) raw 로그와 요약 CSV를 같이 보관

측정 시나리오는 3개입니다.
- 정상 부하: 안정성 기준선 확인
- 버스트 부하: queue full/거절률 관찰
- 장기 요청: timeout/cancellation 반응 관찰

그래프는 최소 3개를 사용합니다.
- throughput 비교 그래프
- p95 latency 비교 그래프
- 503/504 비율 비교 그래프

이 섹션은 발표 전날에 숫자만 갈아끼우는 방식이 아니라, 측정 스크립트까지 고정해서 재현 가능하게 관리하겠습니다.

## 6. 수요코딩회 요구사항 시연 (유지)
시연은 기존 구성 그대로 진행합니다.

첫 번째는 정상 요청입니다.  
`/query`에 정상 SQL을 보내서 200과 정상 data 필드를 확인합니다.

두 번째는 과부하 상황입니다.  
동시 요청을 몰아넣고 `503 + QUEUE_FULL`이 정책대로 나오는지 확인합니다.

세 번째는 timeout/cancellation입니다.  
장기 실행 요청을 보내고 `504` 응답과 취소 경로 동작을 확인합니다.

실패하면 바로 대체 플랜으로 전환합니다.  
예를 들어 과부하가 재현이 안 되면 queue/worker 설정을 더 보수적으로 바꾸고 즉시 재실행합니다.

## 7. 마무리 멘트
정리하면, WEEK8의 핵심 성과는 세 가지입니다.

첫째, API 계약을 먼저 고정해 구현과 테스트 기준선을 맞췄습니다.  
둘째, 스레드풀 + bounded queue로 병렬성과 안정성을 동시에 확보했습니다.  
셋째, timeout/backpressure/cancellation으로 과부하 상황에서도 서버를 제어 가능한 방식으로 동작하게 만들었습니다.
